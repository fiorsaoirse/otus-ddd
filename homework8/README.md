## 1. Разделение моделей (CQRS) для сценария «Просмотр истории заказов в личном кабинете»

## Command-модель

`PlaceOrderCommand` описывает намерение пользователя создать заказ (насколько мне понятно из задания, это должна быть именно DTO). Модель содержит в себе "сырые" данные, НЕ агрегаты, НЕ сущности. В свою очередь handler этой команды уже оперирует не сырыми данными, а преобразовывает их в Aggregate/Entity/Value Object. При этом она не обязана быть удобной для отображения (например, быть плоской или денормализованной).

Пример, как DTO может выглядеть:

```ts
export interface PlaceOrderCommand {
  customerId: string;
  items: Array<{
    productId: string;
    quantity: number;
  }>;
  promoCode?: string;
  deliveryAddress: {
    city: string;
    street: string;
    house: string;
    apartment?: string;
  };
}
```

После получения такой команды handler может:

1. загрузить продукты;
2. создать агрегат `Order`;
3. вызывать методы агрегата (чтобы проверить инварианты);
4. рассчитать стоимость;
5. сохранить заказ.

## Query-модель

`OrderHistoryItemDTO` описывает одну строку истории заказов в личном кабинете.

```ts
export interface OrderHistoryItemDTO {
  orderId: string;
  createdAt: string;
  status: string;
  itemsCount: number;
  itemsPreview: string;
  totalAmount: number;
  totalCurrency: string;
  paymentStatus: string;
  deliveryStatus: string;
}
```

Это плоская денормализованная структура. Она уже содержит все данные, которые нужны UI для отображения строки истории заказов: номер заказа, дату, статус, краткое описание товаров, количество позиций, итоговую стоимость, статус оплаты и доставки.

## Почему для чтения выгоднее отдельная Read-модель

Модель для записи и модель для чтения решают разные задачи.

Модель для записи чаще всего является агрегатом с набором инвариантов, которые должны соблюдаться для бизнес-сущности. Например, заказ должен проверять допустимые статусы, состав позиций, количество товаров, валюту и правила расчета цены. Это важно при изменении состояния, но для чтения истории заказов такая логика не нужна.

Модель для чтения просто отдает данные в удобном для интерфейса виде. Поэтому нет смысла каждый раз собирать сложный агрегат с бизнес-правилами, если пользователь просто открывает список заказов. Более того, моделей для чтения может быть несколько (например, для разных потребителей, для Web приложения или для мобильного приложения модели могут отличаться).

Кроме того, агрегат обычно хранится в нормализованном виде. Чтобы получить строку истории заказов, нужно собрать данные из заказа, позиций заказа, товаров, оплаты и доставки. Если делать это «на лету», каждый запрос на чтение будет вынужден заново выполнять денормализацию. Read-модель позволяет заранее сохранить данные в том виде, в котором они нужны UI.

Модель записи также может содержать поля, которые не участвуют в интерфейсе. Для истории заказов их не нужно запрашивать, потому что они только усложняют чтение и увеличивают объем данных.

Дополнительный плюс разделения command/query моделей — возможность по-разному хранить write model и read model. Здесь речь не о том, что отдельно хранятся сами команды и запросы, а о том, что модель записи и модель чтения могут иметь разные структуры и разные способы хранения.

Write-side обычно хранит данные в форме, удобной для изменения бизнес-состояния: агрегаты, сущности, value objects, нормализованные таблицы и транзакционные ограничения. Например, заказ может храниться через таблицы `orders`, `order_items`, `payments`, `deliveries`.

Read-side может хранить уже готовые DTO/projections в денормализованном виде. Для истории заказов это может быть одна таблица или view `order_history_items`, где одна строка уже содержит `orderId`, дату, статус, количество товаров, краткое описание, итоговую сумму, статус оплаты и доставки.

Физически read-side не обязан быть отдельной базой данных. Это может быть отдельная таблица в той же PostgreSQL, materialized view, read replica, Redis cache, Elasticsearch/OpenSearch индекс или отдельная база, оптимизированная под чтение. Главное — read model оптимизирована под конкретный запрос/экран, а write model оптимизирована под изменение состояния и защиту инвариантов.

Такие read-хранилища обычно проще масштабировать горизонтально (репликация, шардирование). Однако при таком подходе будет другой trade-off — синхронизация между write-side и read-side. Read-модель может обновляться асинхронно, поэтому в системе появляется eventual consistency: после изменения заказа данные в истории могут обновиться не мгновенно.

Еще один недостаток: read-модель обычно собирается и читается как есть. При ее загрузке мы не регидрируем полноценный агрегат и не применяем к нему доменные инварианты. Это нормально для чтения, но означает, что мы доверяем сохраненной read-модели и не используем ее для изменения бизнес-состояния.
