# Homework 7

Задание выполняется на основе карты контекстов из [`homework1`](../homework1/Стратегическое%20проектирование.md).

Система: цифровой тренажер для подготовки к языковому экзамену.

Основные bounded contexts:

- `IdentityContext`;
- `TrainingContext`;
- `AssessmentContext`;
- `StatisticsContext`;
- `RewardsContext`.

## Обоснование паттернов отношений

### Identity -> остальные контексты

Паттерн отношений: `Open Host Service`.

`IdentityContext` является upstream-контекстом для остальных частей системы. Он предоставляет данные об аутентификации и авторизации: идентификатор пользователя и роли, по которым другие контексты определяют доступ к своим возможностям.

У `IdentityContext` много клиентов:

- `TrainingContext`;
- `AssessmentContext`;
- `StatisticsContext`;
- `RewardsContext`;
- другие контексты в будущем.

Поэтому ему важно не подстраиваться отдельно под каждого потребителя, а поддерживать стабильный публичный API. Такой API должен быть простым, понятным и версионируемым.

Модель, которую отдает `IdentityContext`, должна быть минимальной. Остальным контекстам не нужны внутренние детали Identity: устройство, сессия, способ входа, технические поля токена. Им достаточно знать:

- кто пользователь;
- какие у него роли;
- какие действия ему разрешены.

Поэтому `Open Host Service` здесь подходит лучше всего: `IdentityContext` публикует стабильный контракт, а потребители используют его как общий сервис доступа.

### Training -> Assessment

Паттерн отношений: `Customer-Supplier`.

`TrainingContext` и `AssessmentContext` тесно связаны в основном пользовательском сценарии:

1. пользователь проходит задание в `TrainingContext`;
2. `TrainingContext` формирует выполнение задания;
3. выполнение передается в `AssessmentContext`;
4. `AssessmentContext` проверяет ответ и возвращает результат проверки.

В этой паре `TrainingContext` выступает supplier: он предоставляет данные, API и модель выполненного задания.

`AssessmentContext` выступает customer: ему важно, в каком виде `TrainingContext` передаст данные, потому что от этого зависит корректность проверки.

`AssessmentContext` может влиять на контракт `TrainingContext`: например, требовать, чтобы в отправляемой модели были идентификатор задания, тип вопроса, ответы пользователя, критерии проверки или дополнительные metadata.

Это не просто однонаправленное потребление чужой модели. Контексты связаны общим процессом, и downstream-контексту `Assessment` важно участвовать в формировании контракта. Поэтому здесь лучше подходит `Customer-Supplier`, а не `Conformist`.

### Assessment -> Statistics

Основной паттерн отношений: `Anti-Corruption Layer`.

`AssessmentContext` является upstream-контекстом: он производит результат проверки выполненного задания.

`StatisticsContext` является downstream-контекстом: он использует результаты проверки, чтобы строить статистику, графики и прогресс пользователя.

При этом статистика не является ключевым потребителем для `AssessmentContext`. Для Assessment основная задача — проверить выполнение и выдать результат. Ему не важно, как именно Statistics хранит прогресс, какие графики строит и какие агрегаты использует. Более того, `AssessmentContext` может вообще не знать про своих потребителей (при использовании событийной модели в системе).

Из-за этого `StatisticsContext` не имеет сильного влияния на модель `AssessmentContext`.

Чтобы не протащить модель Assessment внутрь Statistics, на стороне `StatisticsContext` лучше использовать `Anti-Corruption Layer`.

ACL будет переводить внешнюю модель результата проверки в собственную модель статистики. Например:

- `TaskSubmission`;
- `Verdict`;
- `Result`;
- `Module`;
- `Level`;

могут быть преобразованы в статистические понятия:

- `ProgressPoint`;
- `AttemptMetric`;
- `ModuleProgress`;
- `LevelProgress`.

Так `StatisticsContext` защищает свою модель и не становится зависимым от терминов и структуры `AssessmentContext`.

Если нет времени или бюджета на полноценный ACL, допустим компромиссный вариант: `Conformist`.

В таком случае `StatisticsContext` будет принимать модель Assessment почти как есть. Это дешевле в реализации, но хуже стратегически: статистика начнет зависеть от изменений в Assessment, и со временем это может привести к сильной связности между контекстами.

Поэтому предпочтительный вариант — `ACL`, а `Conformist` можно рассматривать только как временное упрощение.
