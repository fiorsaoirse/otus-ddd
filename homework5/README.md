## Агрегат

Корень агрегата — `Order`.

Внутри агрегата есть:

- `OrderID`;
- `CustomerID`;
- `status`;
- `maxCount`;
- коллекция `OrderLine`;
- в `OrderLine`: `Product`, `quantity`;
- в `Product`: `ProductID`, `Money`;
- в `Money`: `currency`, `cents`.

## Таблицы

DDL находится в [`db/schema.sql`](db/schema.sql).

Для хранения агрегата используются две таблицы:

- `orders` — корень агрегата `Order`;
- `order_lines` — дочерняя коллекция строк заказа.

`orders.id` хранит `OrderID`, а `orders.customer_id` хранит `CustomerID`.

`order_lines` связана с `orders` через внешний ключ:

```sql
order_id TEXT NOT NULL REFERENCES orders(id) ON DELETE CASCADE
```

Это отражает жизненный цикл внутри агрегата: строки заказа не существуют отдельно от заказа, они не могут изменяться вне заказа.

## Оптимистичная блокировка

В таблице `orders` есть поле:

```sql
version BIGINT NOT NULL DEFAULT 1
```

Оно нужно, чтобы защититься от lost update.

Идея такая: приложение загружает заказ вместе с текущей версией, например `version = 3`. Потом use case меняет агрегат и при сохранении передает в репозиторий ожидаемую версию — `expectedVersion = 3`.

`UPDATE` должен обновить заказ только если в базе все еще лежит та же версия:

```sql
UPDATE orders
SET
    status = $2,
    max_count = $3,
    version = version + 1,
    updated_at = now()
WHERE id = $1
  AND version = $4;
```

Где:

- `$1` — `orderId`;
- `$2` — новый статус;
- `$3` — новый `maxCount`;
- `$4` — версия, которую приложение прочитало до изменения агрегата.

После выполнения запроса репозиторий проверяет количество обновленных строк.

Если обновлена `1` строка, сохранение успешно, а версия заказа стала `expectedVersion + 1`.

Если обновлено `0` строк, значит заказ уже был изменен другим процессом между чтением и сохранением. В этом случае репозиторий должен выбросить ошибку optimistic lock conflict.

Псевдокод:

```ts
const updatedRows = await db.execute(updateOrderSql, [
  order.id,
  order.status,
  order.maxCount,
  expectedVersion,
]);

if (updatedRows === 0) {
  throw new Error("Optimistic lock conflict");
}
```

Строки заказа (`order_lines`) сохраняются в той же транзакции. Версия находится на корне агрегата, потому что именно `Order` является границей согласованности.

## Почему нет отдельной таблицы для OrderLine

Таблица `order_lines` технически отдельная (непосредственно в базе), потому что это коллекция внутри агрегата.

Но с доменной точки зрения `OrderLine` не является самостоятельным агрегатом:

- у нее нет собственного глобального идентификатора;
- она не загружается отдельно от заказа;
- она удаляется вместе с заказом;
- все изменения проходят через `Order`.

Поэтому `order_lines` — это не отдельный агрегат, а часть persistence mapping для `Order`.

## Как сохраняются Value Objects

Value Objects сохраняются как обычные колонки, а не как отдельные таблицы.

Причина: у Value Object нет своего идентификатора и отдельного жизненного цикла. Он существует только как часть сущности или агрегата, который им владеет.

В этой модели:

- `OrderID` сохраняется как `orders.id`;
- `CustomerID` сохраняется как `orders.customer_id`;
- `ProductID` сохраняется как `order_lines.product_id`;
- `Money` сохраняется двумя колонками:
  - `price_cents`;
  - `price_currency`.

Например, `Money` не нужно хранить в отдельной таблице `money`, потому что деньги в строке заказа не являются отдельной сущностью. Это просто значение цены.

Такой маппинг проще поддерживать: данные лежат рядом с владельцем, нет лишних join-ов и нет искусственного identity там, где в доменной модели его нет.

## Почему цена продукта хранится в order_lines

В `OrderLine` хранится `product_id` и снапшот цены:

```sql
product_id TEXT NOT NULL,
price_cents BIGINT NOT NULL,
price_currency currency_code NOT NULL
```

Даже если в системе есть отдельная таблица продуктов, заказ должен хранить цену на момент оформления заказа.

Если цена продукта изменится позже, старые заказы не должны пересчитываться.


## Сага

[Пример реализации](./application/sagas/checkout-saga.ts)

Сага нужна для бизнес-процесса, который нельзя выполнить одной транзакцией в одной базе данных.

В этом примере процесс затрагивает несколько bounded contexts:

- `Orders` — создание и изменение статуса заказа;
- `Payments` — авторизация и списание оплаты;
- `Inventory` — резервирование товаров;
- `Notifications` — отправка уведомлений клиенту.

Если один из шагов падает, сага запускает компенсирующие use case для уже выполненных шагов.

### Тип саги

Я выбрала orchestration saga, мне кажется, что оркестратор проще для дебага процесса.
Хореография позволяет снижать связность, но ценой сложности отладки, а также проще забыть сделать откат в рамках ошибки транзакции - честно говоря, мне кажется, оркестратор надежнее. С учетом, что речь идет об оплате, на мой взгляд лучше использовать именно оркестратора (сложный и важный бизнес-процесс).

### Happy path

Позитивный сценарий:

1. Клиент запускает checkout.
2. `CheckoutSaga` вызывает use case `CreateOrder` в контексте `Orders`.
3. `Orders` создает заказ в статусе `new`.
4. `CheckoutSaga` вызывает `ReserveItems` в контексте `Inventory`.
5. `Inventory` резервирует товары для заказа.
6. `CheckoutSaga` вызывает `AuthorizePayment` в контексте `Payments`.
7. `Payments` успешно авторизует оплату.
8. `CheckoutSaga` вызывает `CapturePayment` в контексте `Payments`.
9. `Payments` списывает деньги.
10. `CheckoutSaga` вызывает `MarkOrderPaid` или `ConfirmOrder` в контексте `Orders`.
11. `Orders` переводит заказ в успешный статус.
12. `CheckoutSaga` вызывает `SendOrderConfirmation` в контексте `Notifications`.
13. `Notifications` отправляет клиенту подтверждение заказа.
14. Сага завершается в статусе `completed`.

### Негативный сценарий: оплата не прошла

Пример: заказ создан, товары зарезервированы, но оплата не прошла.

Шаги:

1. `CreateOrder` успешно создал заказ.
2. `ReserveItems` успешно зарезервировал товары.
3. `AuthorizePayment` или `CapturePayment` завершился ошибкой.
4. `CheckoutSaga` фиксирует ошибку оплаты.
5. `CheckoutSaga` запускает компенсацию `ReleaseReservation` в контексте `Inventory`.
6. `Inventory` снимает резерв с товаров.
7. `CheckoutSaga` запускает компенсацию `CancelOrder` в контексте `Orders`.
8. `Orders` переводит заказ в статус `cancelled`.
9. `CheckoutSaga` вызывает `SendPaymentFailedNotification` в контексте `Notifications`.
10. `Notifications` сообщает клиенту, что заказ не был оформлен из-за ошибки оплаты.
11. Сага завершается в статусе `failed` или `compensated`.

## Идемпотентный обработчик OrderPaidEvent

[Псевдокод обработчика](./application/handlers/order-paid-handler.pseudo.ts)

Задача обработчика: получить `OrderPaidEvent` на стороне `Notifications` и отправить уведомление клиенту, но не отправить его дважды при повторной доставке одного и того же события.

Идея:

1. У каждого события есть уникальный `event.id`.
2. Перед отправкой уведомления handler пытается зарезервировать `event.id` в Redis через операцию вроде `SET key value NX EX ttl`.
3. Если ключ уже есть, значит событие уже обрабатывается или недавно было обработано. Handler ничего не отправляет.
4. Если ключ поставился успешно, handler дополнительно проверяет долговременное хранилище обработанных событий, например таблицу `processed_events` в PostgreSQL.
5. Если событие уже есть в БД, уведомление не отправляется.
6. Если события нет, handler отправляет уведомление.
7. После успешной отправки handler сохраняет `event.id` в БД и оставляет/обновляет ключ в Redis с TTL.
8. Если отправка или запись в БД упала, Redis-резервация снимается, чтобы событие можно было обработать повторно.

Redis здесь используется как быстрый cache/idempotency barrier, чтобы не ходить в БД на каждый быстрый дубль события, операции чтения-записи занимают время и создают дополнительную нагрузку.

PostgreSQL все равно нужен как долговременная защита. Если оставить только Redis с TTL на час, то событие, повторно доставленное через два часа, снова пройдет проверку и уведомление может уйти второй раз.

Условная таблица для долговременного хранения обработанных событий:

```sql
CREATE TABLE processed_events (
    event_id TEXT PRIMARY KEY,
    event_type TEXT NOT NULL,
    processed_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
