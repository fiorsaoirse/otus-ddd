# Value Objects

В домене можно выделить следующих кандидатов на представление их в виде VO:

- Email (вместо string), инварианты:
    - Должен быть не пустым
    - Должен быть корректным (проверка по регулярному выражению)
- QuestionKey (вместо string), инварианты:
    - Должен быть не пустым
    - Должен начинаться с определенного префикса
- VerificationCode (вместо string), инварианты:
    - Должен быть не пустым
    - Должен иметь определенный формат

# Рефакторинг GodUserService

## Проблема исходного варианта

`GodUserService` совмещал слишком много ответственностей:

- проверял и нормализовал email;
- работал с базой данных напрямую;
- создавал пользователя;
- генерировал служебные данные;
- отправлял email;
- писал лог;
- формировал ответ наружу.

Из-за этого класс нарушал SRP: у него было много причин для изменения. Например, изменение правил валидации email, способа хранения пользователя или механизма отправки писем требовало менять один и тот же сервис.

Также нарушался DIP: сервис зависел от конкретных технических деталей, а не от абстракций предметной области.

## Что изменилось

В рефакторинге сервис стал координатором сценария, а не владельцем всей логики.

`Email` выделен в value object. Теперь правила нормализации и валидации email находятся внутри доменного типа:

```go
email, err := domainUser.NewEmail(rawEmail)
```

`User` работает уже с готовым `Email`, а не с сырой строкой. Это делает модель выразительнее: в домене нельзя случайно передать невалидный email как обычный `string`.

Создание пользователя тоже осталось в доменной модели:

```go
newUser := domainUser.NewUser(id, email, now)
```

Изменение email оформлено как поведение пользователя:

```go
user.ChangeEmail(email, now)
```

Так инварианты и изменение состояния остаются внутри доменного объекта.

## Роль UserService после рефакторинга

`UserService` теперь отвечает за orchestration use case:

1. получить и провалидировать входные данные через доменные типы;
2. проверить существование пользователя через `UserRepository`;
3. создать или изменить доменный объект;
4. сохранить результат через `UserRepository`;
5. вызвать `NotificationSender`;
6. записать лог через `Logger`;
7. вернуть DTO наружу.

Сервис не должен сам знать, как именно устроены база данных, SMTP, формат письма или правила парсинга email.

## Какие зависимости стали абстракциями

Вместо прямой работы с инфраструктурой сервис зависит от портов:

- `UserRepository` - хранение и получение пользователей;
- `NotificationSender` - отправка уведомлений;
- `Logger` - логирование.

Это соответствует DIP: высокоуровневый сценарий зависит от абстракций, а конкретные реализации могут находиться в инфраструктурном слое.

## Что осталось за пределами домена

`UserResponse` не является доменной моделью. Это DTO для внешнего слоя. Маппинг из `User` в `UserResponse` должен находиться рядом с application/transport слоем, а не внутри доменного объекта.

Отправка welcome email тоже не является доменной логикой. Домен может сказать, что пользователь создан, но само письмо отправляет внешний адаптер через `NotificationSender`.

## Итог

После рефакторинга код лучше разделяет ответственности:

- домен хранит бизнес-правила и инварианты;
- application service координирует сценарий;
- инфраструктура реализует хранение, уведомления и логирование;
- value object `Email` защищает модель от невалидных примитивов.

Такой вариант проще тестировать, расширять и менять без риска сломать несвязанные части системы.
